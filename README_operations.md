# 補足教材2：HPC で壊れにくい Singularity / PBS 運用

## この教材の目的

この補足教材では、Singularity / Apptainer と PBS を **「一度動かす」だけで終わらせず、実務で壊れにくく運用するための考え方**を整理します。

ハンズオン本編では、以下を体験しました。

- コンテナの中で Python を動かす
- PBS でジョブを投入する
- Array Job で複数入力を処理する

次のステップでは、以下のようなことが重要になります。

- コンテナをどう管理するか
- どのようにバージョンを残すか
- ログや出力をどう整理するか
- 将来、自分や他人が見ても分かる形にするにはどうするか

---

# 1. 一番大事な考え方

まず最初に、一番大事なことを一言で言うと：

> **「動く」だけでは足りず、「後から見ても再現できる」ことが大切**

です。

HPC や研究用途では特に、

- 数週間後
- 数か月後
- 別の利用者
- 別の講義

でも同じように動くことが重要です。

---

# 2. `.sif` を毎回上書きしない

これは非常に大事です。

---

## よくある悪い例

```text
python_env.sif
```

を毎回 build し直して上書きしてしまう。

一見すると便利そうですが、これをやると：

- 昨日まで動いていた環境が消える
- 受講者Aと受講者Bで中身が違う
- 「どの版で実行したか」が分からなくなる

という問題が起きやすくなります。

---

## おすすめの考え方

> **「完成した `.sif` は版付きで残す」**

です。

たとえば：

```text
python_env_handson_v1.sif
python_env_handson_v2.sif
python_env_20260407.sif
python_env_py311_numpy126.sif
```

---

## 初学者向けおすすめ命名

今回の教材なら、まずはこれくらいで十分です。

```text
python_env_handson_v1.sif
```

用途と版が分かれば、かなり管理しやすくなります。

---

# 3. `.def` は必ず残す

これも本当に重要です。

## `.def` は何か？
> **そのコンテナをどう作ったかの設計図**

です。

もし `.sif` だけ残して `.def` を消してしまうと：

- 中に何が入っていたか曖昧になる
- なぜそのライブラリを入れたか分からなくなる
- 将来 build し直せなくなる

ということが起こります。

---

## おすすめ構成

```text
containers/
├── python_env_handson_v1.def
├── python_env_handson_v1.sif
└── README_container.md
```

このように、**`.def` と `.sif` をセットで残す**のがおすすめです。

---

# 4. コンテナ用 README を作る

これも地味ですが、かなり重要です。

例えば以下のような内容を書いておくと便利です。

---

## README に書くとよいこと

- このコンテナの用途
- 含まれている主要ライブラリ
- build 方法
- どの教材 / 解析で使うか
- 更新履歴

---

## 例

```text
用途: PBS + Singularity ハンズオン用
Python: 3.11
ライブラリ: numpy, matplotlib
build日: 2026-04-07
備考: GUIなし環境向けに MPLBACKEND=Agg を設定
```

これがあるだけで、後から見た時の安心感がかなり違います。

---

# 5. PBS スクリプトも「実験条件の一部」

初学者は PBS スクリプトを「ただの実行シェル」と見がちですが、  
実際にはかなり重要な情報を含んでいます。

例えば：

```bash
#PBS -l select=1:ncpus=1
#PBS -l walltime=00:05:00
```

これらは単なるおまじないではなく、

> **「この処理に何CPU必要で、どのくらいの時間を見積もったか」**

を表しています。

つまり PBS スクリプトは、

> **「計算条件の記録」**

でもあります。

---

# 6. ログと出力先を分ける

これも実務でかなり大切です。

初学者がよく困るのは：

- 何が標準出力なのか
- 何がエラーなのか
- 何が解析結果なのか

が混ざって分からなくなることです。

---

## おすすめディレクトリ構成

```text
project/
├── input/
├── output/
├── logs/
├── run/
└── containers/
```

たとえば：

- `input/` → 入力データ
- `output/` → 解析結果
- `logs/` → PBSログ
- `run/` → 実行スクリプト
- `containers/` → `.sif`, `.def`

のように分けると整理しやすいです。

---

# 7. PBS のログを意識する

PBS ではジョブの標準出力や標準エラーがファイルとして残ります。

例えば：

```bash
#PBS -j oe
```

は、

> **標準出力と標準エラーをまとめる**

設定です。

---

## さらに整理したい場合

環境によっては以下のようにログ出力先を分けられます。

```bash
#PBS -o logs/
#PBS -e logs/
```

これを使うと、ジョブの痕跡を追いやすくなります。

---

# 8. Array Job は「大量処理の基本形」

HPC では、以下のような処理が非常によくあります。

- 100個のCSVを同じ解析にかける
- 500枚の画像を同じ前処理にかける
- 1000個の音声ファイルを特徴抽出する
- 多数の条件でシミュレーションを回す

こういう時の基本形が：

> **「1入力 = 1ジョブ」**

です。

PBS の Array Job は、それを効率よく扱うための仕組みです。

---

## 例

```bash
#PBS -J 1-100
```

これは：

> **100個の似たジョブをまとめて投げる**

ための指定です。

---

# 9. CPU を増やしても速くなるとは限らない

これは並列計算機の初学者が非常に引っかかりやすいポイントです。

例えば：

```bash
#PBS -l select=1:ncpus=8
```

と書いたとしても、プログラム側が1スレッドでしか動かないなら、  
**8 CPU を予約しただけで、処理はほとんど速くならない**ことがあります。

---

## ここで大事な理解

### PBS の役割
> **資源（CPU, メモリ, 時間）を割り当てる**

### プログラム側の役割
> **その資源を実際に使う**

つまり：

> **「PBS は並列化してくれる道具ではなく、並列化された処理を実行しやすくする道具」**

です。

これは非常に重要です。

---

# 10. 実務でおすすめの運用ルール

最後に、初心者のうちから身につけておくと役立つルールをまとめます。

---

## おすすめルール

### 1. コンテナは版付きで保存する
例：

```text
python_env_handson_v1.sif
```

---

### 2. `.def` を必ず残す
例：

```text
python_env_handson_v1.def
```

---

### 3. ジョブスクリプトも保存する
例：

```text
run_pbs.sh
run_array.sh
```

---

### 4. ログと結果を分ける
例：

```text
logs/
output/
```

---

### 5. ディレクトリ構成を揃える
例：

```text
input/
output/
run/
containers/
```
